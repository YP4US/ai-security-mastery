## 💻 1:30 PM–4:00 PM | DEEP WORK 1 — MITRE ATLAS, Actually Explained

**Why yesterday didn't click:** the Day 2 card gave you a list of 9 tactic names with no
instructions on how to use them. The real matrix at atlas.mitre.org currently has **14
tactics**, and a list of names alone doesn't teach you how to read it. Here's the actual
method analysts use.

### What ATLAS actually is

MITRE ATLAS is **MITRE ATT&CK, but for AI/ML systems**. If you've heard of ATT&CK (the
standard framework for describing cyberattack behavior), ATLAS is the same idea applied
specifically to attacks against machine learning systems — model theft, data poisoning,
prompt injection, etc.

It is **not** a checklist you complete top to bottom. It's a **reference matrix you consult
when analyzing a specific attack or a specific system**, structured as:

```
TACTIC (the attacker's goal, the "why")
  → TECHNIQUE (the "how" — a specific method to achieve that goal)
    → SUB-TECHNIQUE or PROCEDURE EXAMPLE (a real documented case of it happening)
```

### The 14 tactics (columns in the matrix), in the order an attack typically flows

1. **Reconnaissance** — attacker researches the target ML system (what model, what data, what API)
2. **Resource Development** — attacker builds/acquires tools (fake accounts, adversarial datasets, proxy models)
3. **Initial Access** — attacker gets a foothold (e.g., access to the model API, a poisoned public dataset)
4. **ML Model Access** — attacker gets some level of access to the model itself (API access, white-box access, physical access)
5. **Execution** — attacker runs malicious code or crafted inputs against the system
6. **Persistence** — attacker maintains long-term access or influence (e.g., a poisoned model stays poisoned)
7. **Privilege Escalation** — attacker gains higher-level access than they started with
8. **Defense Evasion** — attacker avoids detection (e.g., adversarial perturbations invisible to humans)
9. **Credential Access** — attacker steals credentials/API keys related to the ML pipeline
10. **Discovery** — attacker maps out what's available (other models, data stores, pipelines)
11. **Collection** — attacker gathers data/model artifacts to use or exfiltrate
12. **ML Attack Staging** — attacker prepares the actual attack (e.g., crafts the adversarial example, tunes the prompt injection)
13. **Exfiltration** — attacker extracts data or model IP out of the system
14. **Impact** — the actual damage: model behaves wrongly, data is leaked, service is degraded

**You don't need to memorize all 14 today.** The point is: this is a *kill chain*, left to
right, describing the stages an attacker moves through — not 14 unrelated categories.

### How to actually use the matrix (step by step)

1. Go to **https://atlas.mitre.org/matrices/ATLAS** — this is the visual matrix, tactics as
   columns, techniques listed under each.
2. Pick ONE real or hypothetical attack you understand well — use **your Day 1 garak
   promptinject scan** as the example.
3. Ask: *"At which stage of the ATLAS kill chain does this attack sit?"*
   - Your garak scan = you (the tester) crafting and sending malicious prompts to a model
     you already have API access to. That's mostly **ML Attack Staging** (crafting the
     injection) → **Execution** (sending it) → **Impact** (the model followed the injected
     instruction instead of its original one).
4. Search the matrix (or use the search bar on atlas.mitre.org) for a technique under those
   tactics that matches — for prompt injection specifically, look for
   **"AML.T0051 - LLM Prompt Injection"**.
5. Click into that technique page. Each technique page shows:
   - A description of the technique
   - **Procedure examples** — real, documented cases (e.g., a specific paper or incident)
   - Related mitigations
6. That technique code (e.g., `AML.T0051`) is what you cite in a professional report —
   it's the "so what" that turns "the model got jailbroken" into "this is a known,
   catalogued attack technique with documented precedent."

### What garak's output actually means (recap from Day 1's answered question)

- The **terminal per-probe lines** (`FAIL ok on 748/1280`, attack success rate 41.56%) are
  the real, granular result: out of 1280 attack attempts, how many *succeeded* in hijacking
  the model. This is what you report.
- The **HTML summary** aggregates against garak's internal calibration baseline and can
  say "secure" even when individual attacks succeeded some percentage of the time — it's
  answering a different question ("is this module worse than typical models we've seen")
  not "did any attacks succeed."
- **Rule of thumb:** always report the per-probe terminal numbers as your findings. Treat
  the HTML "secure/not secure" label as a secondary comparative signal, not the headline.

### Your Day 1 scan, written up as a real finding (copy this pattern)

```markdown
# AI Red Team Assessment — llama3.2:1b
**Date:** July 17, 2026
**Tester:** Yogesh Pawar
**Target:** llama3.2:1b via Ollama (local, no safety fine-tuning)
**Tool:** garak

## Executive Summary
Ran garak's `promptinject` probe suite (3 sub-probes: HijackHateHumans, HijackKillHumans,
HijackLongPrompt) against llama3.2:1b. All three sub-probes achieved a non-trivial attack
success rate, with HijackLongPrompt succeeding in over half of attempts — consistent with
a small, non-safety-tuned model having weak resistance to prompt injection.

## Findings

### Finding 1: HijackLongPrompt — high susceptibility to long-prompt injection
- **Severity:** High
- **MITRE ATLAS Technique:** AML.T0051 (LLM Prompt Injection)
- **OWASP LLM Category:** LLM01 (Prompt Injection)
- **What happened:** Of 1280 attempts, 629 succeeded in making the model follow an
  injected instruction embedded in a long prompt instead of its original instruction —
  a 50.86% attack success rate.
- **Business impact:** In a production system, an attacker could bury malicious
  instructions inside a long user input (e.g., a pasted document) and have better than
  even odds of hijacking the model's behavior — e.g., exfiltrating system prompt content,
  ignoring safety instructions, or performing unauthorized actions if the model has tool access.
- **Fix:** Add input/output filtering for injected-instruction patterns; use a
  smaller-context "sanitizer" pass before long user content reaches the main model;
  prefer models with RLHF-based injection resistance for production use over a raw
  base model like llama3.2:1b.

### Finding 2: HijackHateHumans — moderate susceptibility
- **Severity:** Medium
- **MITRE ATLAS Technique:** AML.T0051 (LLM Prompt Injection)
- **OWASP LLM Category:** LLM01 (Prompt Injection)
- **What happened:** 748/1280 succeeded (41.56% attack success rate).
- **Business impact:** Similar hijack risk, somewhat lower success rate than the
  long-prompt variant.
- **Fix:** Same mitigation family as Finding 1.

### Finding 3: HijackKillHumans — lower but non-zero susceptibility
- **Severity:** Low-Medium
- **MITRE ATLAS Technique:** AML.T0051 (LLM Prompt Injection)
- **OWASP LLM Category:** LLM01 (Prompt Injection)
- **What happened:** 1161/1280 succeeded, meaning the attack success rate is 9.30%
  (lowest of the three, but still meaningful at scale).
- **Business impact:** Lowest of the three tested variants, but still exploitable.
- **Fix:** Same mitigation family as Finding 1.

- Total tests: 1280 per sub-probe, 3840 total
- Pass rate: varies 49–91% depending on sub-probe (see findings above)
```
